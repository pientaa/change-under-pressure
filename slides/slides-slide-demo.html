Redefining Legacy

Legacy ≠ old tech.

Legacy is rising cost of change.

Age is irrelevant. Friction is everything.

The Cost Drivers

Technical Coupling

  • Shared domain models
  • Hidden decisions inside orchestration
  • Missing testing seams
  • Highly intertwined logic

Coordination Tax

  • Blurred ownership boundaries
  • Cross-team pull requests
  • Mandatory sync meetings
  • Extreme cognitive load

03 First Seam: Fulfillment API

Extract priority logic behind a contract boundary.

04 Behind the Seam: PriorityService

The Fulfillment team implements the logic behind FulfillmentApi.

PriorityService.kt 
class PriorityService : FulfillmentApi {

  companion object {
    private val PRIORITY_THRESHOLD = Money("100.00")
    private val GOLD_PRIORITY_THRESHOLD = Money("50.00")
  }

  override fun determinePriority(input: PriorityInput):
    PriorityDecision {
    if (input.vip)
      return priorityRouting(NEXT_DAY)

    if (input.segment == GOLD
      && input.campaignCode == "PARTNER2026"
      && input.finalPrice >= GOLD_PRIORITY_THRESHOLD
      && input.shippingMethod != ECONOMY
    ) return priorityRouting(EXPEDITED)
                                        ▸
…continued 
    if (input.finalPrice >= PRIORITY_THRESHOLD
      && input.shippingMethod != ECONOMY
    ) return priorityRouting(EXPEDITED)

    return standardRouting()
  }

  private fun priorityRouting(sla: SlaTier) =
      PriorityDecision(priority = true, slaTier = sla)

  private fun standardRouting() =
      PriorityDecision(priority = false,
          slaTier = SlaTier.STANDARD)
}

05 Behind the Seam: PricingService

The Pricing team implements the logic behind PricingApi.

PricingService.kt (initial extraction) 
class PricingService(
    private val campaignCatalog: CampaignCatalog
) : PricingApi {

  override fun calculate(cmd: PricingCommand): PricingResult {
    val campaign = cmd.campaignCode?.let {
        campaignCatalog.findCampaign(CampaignCode(it))
    }
    val discount = campaign?.discountPercent ?: BigDecimal.ZERO

    return if (discount > BigDecimal.ZERO) {
        val factor = BigDecimal.ONE - discount / BigDecimal(100)
        PricingResult(
            finalPrice = cmd.basePrice * factor,
            appliedDiscount = true,
            discountName = campaign?.description
        )
    } else {
        PricingResult(cmd.basePrice, appliedDiscount = false)
    }
  }
}

08 Decouple Data Ownership

Split the model. Drop cross-context writes. Each context persists its own data.

PlaceOrderService.kt (refactored) 
class PlaceOrderService(
    private val customerApi: CustomerApi,
    private val pricingApi: PricingApi,
    private val fulfillmentApi: FulfillmentApi,
    private val checkoutRepo:
        CheckoutOrderRepository
) {
  fun placeOrder(order: CheckoutOrder):
      PlaceOrderResult {
    val profile = customerApi.findProfile(order.customerId)

    val pricingCmd =
        CheckoutToPricingCommandMapper.map(order)
    val pricingResult = pricingApi.calculate(pricingCmd)

    val priorityInput =
        CheckoutToPriorityInputMapper
            .map(order, profile, pricingResult)
    val decision =
        fulfillmentApi.determinePriority(priorityInput)

    order.status = CONFIRMED
    checkoutRepo.save(order)

    return PlaceOrderResult(order, profile,
        pricingResult, decision)
  }
}

Legacy Modernization Techniques

Everything we just did has names — from Working Effectively with Legacy Code and Domain-Driven Design.

"A seam is a place where you can alter behavior without editing in that place."

— Michael Feathers, Working Effectively with Legacy Code

Feathers identifies different seam types — places where you can vary behavior — and dozens of dependency-breaking techniques to get tangled code under test. Let me show you the ones that matter most.

🔌 Object Seam

Extract a contract — swap implementations freely. Production vs. test, same interface.

Production — real dependency 
interface CampaignCatalog {
  fun findCampaign(code: CampaignCode): Campaign?
}

class DatabaseCampaignCatalog(private val db: DataSource) : CampaignCatalog {
  override fun findCampaign(code: CampaignCode) = db.query("SELECT … WHERE code = ?", code)
}
Test — fake at the seam 
class FakeCampaignCatalog : CampaignCatalog {
  private val campaigns = mutableMapOf<CampaignCode, Campaign>()

  fun add(code: CampaignCode, c: Campaign) { campaigns[code] = c }

  override fun findCampaign(code: CampaignCode) = campaigns[code]  // no DB needed
}

🧬 Subclass & Override Method

Override in tests to break dependencies without changing production code.

Hard-to-test production code 
open class NotificationSender {

  fun processOrder(order: Order) {
    // business logic we want to test
    val msg = buildMessage(order)
    sendEmail(msg)  // side effect!
  }

  open fun sendEmail(msg: String) {
    emailClient.send(msg)
  }
}
Test — override the seam 
class TestableNotificationSender
    : NotificationSender() {

  val sent = mutableListOf<String>()

  override fun sendEmail(msg: String) {
    sent.add(msg)  // capture, don't send
  }
}

// Now we can test processOrder
// without actually sending emails

📦 Parameterize Method

Push a dependency to the caller — pass behavior as a parameter.

Before — hard-coded dependency 
fun applyDiscount(order: Order) {
  val rate = lookupRate(order.customerId)
  order.finalPrice = order.basePrice * (1 - rate)
}

// lookupRate talks to the database — Can't test without full infra
After — pass it in (default = production) 
fun applyDiscount(order: Order, rateLookup: (String) -> BigDecimal = ::lookupRate) {
  val rate = rateLookup(order.customerId)
  order.finalPrice = order.basePrice * (1 - rate)
}

// Test: applyDiscount(order) { "0.10".toBigDecimal() }
// Prod: applyDiscount(order) — uses default
